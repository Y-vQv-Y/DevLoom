# DevLoom 用户操作手册

本文适用于 DevLoom 私有部署的管理员、团队成员和移动端用户。界面文字可能因语言或版本略有不同，但操作链路保持一致。部署和运维请参阅 [部署手册](./DEPLOYMENT_CN.md)。

## 1. 使用前检查

请向管理员确认以下内容：

- Web 服务地址，例如 `https://devloom.intra.example`；
- 登录邮箱和初始密码；
- 至少一个可用模型和开发镜像；
- 至少一个状态为“在线”的私有开发主机；
- Git 仓库地址和具有相应权限的个人访问令牌（PAT）。

只有管理页面可用、但缺少 Taskflow/runner、在线主机、模型或镜像时，无法运行 AI 任务、远程终端、文件管理和在线预览。

## 2. 登录与账号

### Web/桌面端

1. 在浏览器打开服务地址，进入“登录”。
2. 输入管理员分配的邮箱和密码。
3. 阅读并同意用户协议、隐私政策和 AI 数据处理说明。
4. 首次登录后打开左下角账号入口，修改初始密码并检查通知设置。

桌面端使用与 Web 相同的账号和服务。若页面反复回到登录页，检查系统时间、Cookie、HTTPS 证书和反向代理配置。

### Android/iPhone

1. 安装管理员提供的 APK/IPA；iPhone IPA 必须包含当前设备或来自 TestFlight/App Store。
2. 在登录页点“配置服务地址”，填写完整 `https://...` 地址，不要追加 `/api`。
3. 仅当企业入口启用了 Basic Auth 时填写 `user:password`。
4. 同意协议后，用同一邮箱和密码登录。

移动端默认不启用 Apple 登录；密码登录不依赖套餐、充值或 Expo 账号。

## 3. 管理员首次初始化

首次部署通过 `DEVLOOM_INIT_TEAM_EMAIL`、`DEVLOOM_INIT_TEAM_PASSWORD` 和 `DEVLOOM_INIT_TEAM_NAME` 创建初始团队。管理员登录后依次完成以下配置。

### 成员与权限

进入“管理后台 → 成员”：

1. 添加成员并设置角色、状态和团队分组。
2. 将主机、镜像、模型等资源授权给对应分组。
3. 停用离职账号并轮换其 Git Token、模型密钥和共享终端链接。

不要让普通成员共用管理员账号。当前商业席位和企业 License 默认关闭，不影响开源团队成员管理。

### 接入模型

普通用户可在“控制台 → 设置 → 模型”维护自己的模型；管理员也可在“管理后台 → 设置”维护团队模型。

1. 点击“添加模型”。
2. 选择供应商或兼容接口。
3. 填写 API Base URL、API Key、模型名和备注。
4. 按服务实际协议选择 `openai_chat`、`openai_responses` 或 `anthropic`。
5. 执行健康检查，确认后再授权团队使用。

内网部署应填写内网模型网关地址。Ollama 等本地模型需要允许后端网络访问，不能填写只对用户电脑有效的 `localhost`。

### 添加开发镜像

进入“控制台 → 设置 → 镜像”或“管理后台 → 设置 → 镜像”：

1. 添加完整 OCI 镜像引用，例如 `registry.intra.example/devbox/ubuntu-22.04:2026.07`。
2. 填写清晰备注，并按需设为默认镜像。
3. 确保所有开发 host 能登录镜像仓库并拉取该镜像。
4. 将镜像授权给需要的团队分组。

DevLoom 不负责生成 Ubuntu、Debian 或其他开发镜像。镜像必须与所采用 runner/Taskflow 的约定兼容，且包含任务所需的 shell、Git 和开发工具。

### 安装并注册开发主机

进入“控制台 → 设置 → 主机”：

1. 点击添加主机，复制系统生成的安装命令。
2. 在准备用于运行开发环境的 Linux 主机上，以受控管理员身份执行。
3. 等待主机出现在列表并显示“在线”。
4. 修改备注，检查 IP、架构和 VM 数量，并授权给团队分组。

离线部署时安装命令会读取 `/static/installer/<arch>/` 下的离线包；文件由部署方提供。不要在未审查脚本、来源和权限的主机上执行安装命令。

## 4. 绑定 Git 身份

进入“控制台 → 设置 → Git 身份”。移动端入口为“个人 → Git 账号”。

1. 在 GitHub、内部 GitLab、Gitee、Gitea 或其他支持平台创建 PAT。
2. 点击“添加身份”，选择平台。
3. 填写实例 Base URL，例如 `https://gitlab.intra.example`，以及用户名、邮箱、Token 和备注。
4. 保存后确认身份可列出或访问目标仓库。

建议最小权限：读取仓库；如需让任务创建分支、提交或合并请求，再授予写仓库/API 权限；如需自动审查，再授予管理 webhook 的权限。禁止使用管理员全局 Token 代替个人身份。

Git OAuth 快捷绑定默认关闭，手工 PAT 始终可用。企业自签名 GitLab 应由管理员安装内部 CA，不建议跳过 TLS 校验。

## 5. 创建项目

项目把 Git 身份与一个仓库关联起来。Web 端可从侧栏项目区新建，移动端进入“项目 → 新建项目”。

1. 选择已绑定的 Git 身份。
2. 选择或输入仓库地址。
3. 填写项目名称和可选说明。
4. 保存并打开项目，确认仓库信息、任务列表和 Issue/MR 数据可以读取。

一个仓库可以产生多个任务。若仓库无法访问，优先检查 Token 权限、Base URL、仓库 URL、内部 CA 和 DNS，而不是重复创建项目。

## 6. 创建 AI 开发任务

Web 端进入“任务”并点击新建，或在项目详情中发起任务。移动端点击“任务”页右下角 `+`。

1. 选择项目/仓库，也可手工输入仓库 URL、上传 ZIP 或不关联仓库快速开始。
2. 选择模型。
3. 选择在线私有开发主机和开发镜像。
4. 按需选择技能、插件或 MCP 工具。
5. 输入明确指令，包括目标、限制、期望验证和不能修改的范围。
6. 同意 AI 数据处理提示并提交。

示例：

```text
修复登录页 Token 刷新失效问题，只修改认证模块；补充回归测试，运行相关测试并说明结果，不升级依赖。
```

任务会依次拉取镜像、创建环境、启动环境并连接 Agent。长时间停在“正在准备开发环境”时，查看条件详情并联系管理员检查 host、镜像仓库、Taskflow 和资源容量。

## 7. 任务工作区

### 对话与控制

- 在输入框继续补充需求、回答 Agent 问题或上传附件。
- 运行期间可切换可用模型；切换只影响后续请求。
- 任务达到并发上限时，先终止不再需要的运行中任务再重试。
- “终止”停止当前任务；“删除”会移除记录且通常不可恢复，删除前先下载所需文件或提交代码。

AI 输出可能不完整或错误。合并代码前必须人工审查 diff、测试、安全影响和许可证。

### 文件与代码变更

在任务详情打开“文件/变动”：

- 浏览工作区目录和文件内容；
- 查看单个文件 diff；
- 下载文件、目录 ZIP 或完整项目；
- 确认敏感文件、密钥和大文件没有被意外加入提交。

下载依赖开发环境仍然在线。重要结果应尽快提交到 Git 或下载到受控存储。

### 远程终端

Web 任务详情可打开终端；也可在“设置 → 虚拟机”中进入独立环境。终端命令直接在远程开发环境执行：

```bash
git status
git diff --stat
go test ./...
pnpm test
```

只运行当前项目实际支持的命令。终端拥有环境内权限，不要粘贴来源不明的命令、生产密钥或长期凭据。

### 在线预览

当应用在开发环境监听端口后，任务页显示“在线预览”：

1. 让开发服务监听 `0.0.0.0`，不要仅监听 `127.0.0.1`。
2. 点击“在线预览”，选择检测到的端口。
3. 移动端可在内置浏览器查看、收起并重新展开。

未显示端口时检查应用是否仍在运行、监听地址、preview 服务、动态端口防火墙和访问 URL。

## 8. 独立云开发环境

云开发环境是手工创建的远程 VM/容器，不必先创建 AI 任务。

### Web

进入“控制台 → 设置 → 虚拟机”，选择主机、镜像、模型、CPU/内存和生命周期，然后创建。创建成功后可打开终端、查看状态或删除环境。

### 移动端

进入“个人 → 开发环境”：

1. 填写环境名称。
2. 选择在线主机、镜像和模型。
3. 点击“创建”。当前移动端默认请求 2 核 CPU、8 GB 内存和 3 小时生命周期。
4. 环境就绪后点“打开终端”，可发送命令、刷新连接或发送 `Ctrl-C`。
5. 用完后删除环境，释放计算资源。

可选操作系统由管理员添加的兼容开发镜像决定；界面本身不会自动构建 Ubuntu、Debian、Rocky Linux 等镜像。

## 9. GitHub/GitLab `@devloom` 自动审查

此功能由开源后端实现，但默认关闭。管理员必须先完成：

1. Web 编译变量 `VITE_ENABLE_AUTO_REVIEW=true`。
2. 后端 `review_agent.model_id` 和 `review_agent.image`。
3. `task.core`、`task.memory` 以及至少一个在线私有 host。
4. GitLab 实例、PAT、TLS、webhook secret 和仓库 webhook 权限。
5. 项目中显式开启“自动审查”。

启用后，在 GitLab 合并请求评论中输入：

```text
@devloom 请审查这个合并请求，重点检查鉴权、数据泄露和回滚风险。
```

默认触发词由 `task.at_keyword` 配置。没有响应时检查 webhook 投递记录、项目绑定身份、评论事件、模型/镜像 ID、host 状态和 backend 日志。机器人建议仅供参考，不能代替正式代码评审。

GitHub Pull Request 评论也支持 `@devloom review`、`@devloom fix 修复内容`、`@devloom explain 解释内容` 和 `@devloom plan 计划内容`。`fix` 只允许在任务隔离分支中提交；能否自动创建 PR/MR 取决于兼容运行时是否执行工作区策略，DevLoom 不会自动合并保护分支。

已经启用自动审查的 GitHub 项目需要先关闭再重新开启一次，使仓库 Webhook 增加 `issue_comment` 事件订阅。

## 10. 技能、插件与 MCP

- “设置 → 工具与 MCP”用于添加个人工具；“管理后台 → 技能/MCP”用于团队级资源。
- 私有部署的默认技能列表为空，不会自动连接外部插件仓库。
- 安装并授权技能后，可用 `VITE_DEFAULT_SKILL_IDS` 和 `EXPO_PUBLIC_DEFAULT_SKILL_IDS` 在重新编译客户端时设置默认选择。
- MCP Server 可能读取代码或访问内网服务，只添加可信地址和最小权限 Token。

## 11. 移动端远程开发能力

移动端支持：

- 查看、创建、筛选、终止和删除任务；
- 创建项目并绑定 Git 身份；
- 添加和检查自定义模型；
- 选择主机/镜像创建开发环境；
- 远程终端、任务文件、diff、下载和端口预览；
- 在任务中继续对话、上传图片和切换模型。

复杂代码编辑和大规模 diff 审查仍建议使用 Web/桌面端。移动端网络切换后若 WebSocket 断开，返回任务页或点刷新重新连接，不要连续重复创建任务。

## 12. 商业功能与数据说明

套餐、充值、积分购买、支付、签到、邀请、社区广场、企业 License、Git OAuth 快捷绑定和 Apple 登录默认关闭。关闭这些功能时，密码账号、手工 Git PAT、模型、项目、任务、主机、云开发环境、终端、文件和预览仍可正常使用。

任务指令、代码、文件和图片会发送到所选模型服务商。私有部署并不自动保证模型数据留在内网；是否出网取决于模型 Base URL、MCP、Git 平台、通知和管理员网络策略。提交敏感代码前确认组织的数据处理要求。

## 13. 常见问题

| 问题 | 处理方法 |
|---|---|
| 登录失败 | 确认服务地址、邮箱/密码、协议勾选、系统时间和证书 |
| 没有模型 | 管理员或用户先在“设置 → 模型”添加并通过健康检查 |
| 没有可选主机 | 安装兼容 host，并确认状态在线且已授权当前分组 |
| 没有可选镜像 | 添加兼容 OCI 开发镜像并授权给当前用户/分组 |
| 仓库不可访问 | 检查 PAT、仓库 URL、实例 Base URL、CA、DNS 和权限 |
| 环境创建失败 | 检查 host 容量、镜像拉取、Taskflow 和 CPU/内存限制 |
| 终端连接失败 | 环境必须在线；检查 WebSocket、反向代理、证书和网络 |
| 看不到在线预览 | 服务需监听 `0.0.0.0`，并确保 preview 与动态端口可达 |
| 自动审查无响应 | 检查功能开关、项目 webhook、评论触发词、模型、镜像和 host |
| iPhone 无法安装 IPA | 描述文件需匹配 Bundle ID、Team ID、证书及设备 UDID |

联系管理员时提供任务 ID、发生时间、页面、操作步骤、错误文本和可脱敏截图。不要发送密码、PAT、模型 API Key、Cookie、证书私钥或完整环境配置。
